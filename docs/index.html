General Info

Real-Time Edge Wildfire Detection System

Project Overview

The system detects wildfire and smoke events from a live camera stream on an NVIDIA Jetson Orin Nano. A vision-language model describes the scene, a semantic text classifier decides whether the description indicates wildfire or smoke, and the system can trigger alerts and save evidence clips.

Why Edge AI?

  • Wildfires must be detected early, before a small event becomes harder to contain.
  • Satellites can miss small early fires or be limited by update frequency and resolution.
  • Human watchtowers require continuous manpower.
  • Edge devices can be placed closer to the monitored area.
  • Local processing reduces cloud dependence and latency.
  • Human verification can still be used after the system saves an evidence clip.

Camera

Captures the monitored scene as RGB video input.

VLM Container

Samples frames and generates short text descriptions of the scene.

UDP

Passes short time-sensitive messages with low latency.

Classifier Container

Uses semantic classification to compare text against wildfire and no-wildfire hypotheses.

Alert Logic

Reduces false alerts by requiring repeated wildfire decisions in a time window.

Circular Buffer

Keeps recent video so the system can save moments before and during an event.

Why These Choices?

Why a camera?

It is affordable, scalable, and gives visual evidence that can be checked by a human.

Why a VLM?

Early fire may appear as smoke, glow, haze, distant flame, or context. A VLM describes the whole scene instead of only looking for one object shape.

Why MNLI?

The VLM output is text. MNLI lets the system test whether that text supports a wildfire or smoke hypothesis without training a custom classifier from scratch.

Why UDP?

Messages are short and time-sensitive. A fresh scene description is more useful than a delayed backlog.

Why Docker?

The VLM, classifier, and monitoring tools can be updated or debugged separately.

Why fast storage?

The system handles logs, rolling buffers, evidence clips, and swap. Slow storage can create delays or instability.

Repository and Run Command

SD Express Student Challenge Report

Real-Time Wildfire Detection

This section summarizes the storage-focused student challenge work: how microSD Express and other fast removable storage can support a real-time edge AI workload on a constrained Jetson device.

Challenge Focus

The report focuses on storage as part of the complete AI pipeline, not only model inference. The workload includes video buffering, logs, clips, runtime data, and swap pressure.

Why Storage Matters

Rolling buffers and evidence clips require continuous writes. Logs and model/runtime data create additional access. Swap can become important when Jetson RAM is constrained.

System Summary

The test setup combines a camera, Jetson Orin Nano, Dockerized VLM and classifier services, semantic wildfire/no-wildfire decisions, alerts, and evidence capture.

Results Summary

Tests used recorded wildfire footage and live camera input. Under the tested workload, the storage graphs suggest that fast removable storage can support logging, buffering, and swap activity without becoming the only bottleneck.
